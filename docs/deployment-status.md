# Deployment Status — single source of truth

**This is the authoritative record of what is LIVE for each reader edition.** Do not infer deployment state from a per-repo `CLAUDE.md`, a README, or a memory — those have drifted stale. When in doubt, the tiebreaker is the **git log of the live corpus directory** in each repo (`git log -- <live-layer-path>`). Update THIS file whenever a corpus is re-deployed.

## All three readers run the mechanical-first pipeline (deploy-then-refine)

Every live edition is **mechanical-first method output** (v0 → v1 clause-atoms → v1.5 binding rules), deployed and then refined. The hand-edited editions were **superseded** — they are NOT a protected "sacred" layer and NOT a deploy oracle. The deploy bar is **"new AND superior by the method's own bidirectional-test yardstick,"** never "as good as the old hand-edits." (See `feedback_hand_edit_is_a_datapoint`.)

**What *is* still sacred:** the **source text** — v0 word forms, consonants/niqqud/te'amim (Hebrew), punctuation, verse refs. Never modified. The ATU **line-breaks/segmentation** are regenerable method output, not hand-edits to protect.

## Pipeline stages (framework §3) — there is NO v4

`v0` source → `v1` treebank clause-atoms → `v1.5` binding rules (**the deployed mechanical-first draft**) → `v2` *optional* narrow-task LLM adjudication on residuals → `v3` editorial review. No corpus has run a real v2-LLM stage yet; mechanical-first deploy = v0→v1→v1.5. (Directory-naming has drifted per repo — see `reference_corpus_pipeline_map`; the *stage* is what this table's "built from" column normalizes against.)

## Why this file exists

Deployment state was scattered across git history + a partial memory + stale per-repo orientation docs, which repeatedly produced the false "the live edition is a hand-edited sacred layer" assertion. This file is the one place to check, and the per-repo docs should point here rather than restate (and re-stale).
